# The Oven — [Project Name]

## How to work with this document (fixed explanation — do not delete)

This document is called "the oven", and it is the heart of a project managed
with the STUDIO method — a method for planning and building processes and
systems with AI, for people who are not programmers.

The idea behind the method: a whole project is not one big task but a
process — stages that depend on each other. So we do not build right away:
we pass through six stations, each answering one question, and everything
that becomes clear in them is recorded here, in this document. "The oven" —
because this is where everything cooks: whoever reads the document sees the
whole project.

The six stations:
1. Scout — what do I not yet know? Spread the dream into parts, mark the
   gaps, and send out research questions. Also look for existing tools and
   solutions.
2. Target — what do I really want? Dig beneath the first idea ("what else
   would I want?" again and again), and phrase an axiom: "It cannot be
   that... / I must have..." — the value the project protects, which
   disqualifies any solution that contradicts it.
3. Unify — what are the functional units? Cut the project into units, each
   doing one clear thing and forming a closed content world. For each unit,
   choose the right worker: a human / an AI agent / a skill or automation.
4. Design — how does everything look and feel? One language for all the
   outputs, based on references. No adjectives without a reference.
5. Index — what connects to what, and what is the order? Each unit gets a
   signature: what it receives and what it returns. The build order derives
   from the dependencies, and every connection gets a stopping point.
6. Operate — where do we start and with what do we build? If there is a gap
   the project cannot exist without — test exactly that first, small and
   cheap (feasibility). If there is none — build the leanest version that
   works end to end (a trial run), and only then expand.

The rules of the road:
- Thinking apart, execution apart: do not execute in the window where the
  thinking happened. From the thinking window, distill a clean instruction
  for a fresh execution window.
- Broke? Fix the factory: diagnose (prompt / context / tool / model), and
  change one variable only. A failure gives birth to a rule; a success that
  repeats becomes an asset (a skill).
- Mini-studio: a unit that is too big gets a small studio of its own. Stop
  cutting when every piece has one self-standing document that someone with
  no background could execute from.

The folder: next to this document there are (or will be) subfolders for the
stations — Scout for research findings, Target for the plan document, Design
for references, Operate for build plans. The "folder map" below records what
sits where.

If you are an AI reading this document: first tell the project owner where
the project stands — what is full, what is missing, and what the natural
next step is. Update this document at the end of every piece of work, and
always keep it faithful to the real state of the project.

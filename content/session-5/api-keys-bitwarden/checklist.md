# צ'קליסט אבטחת מפתחות API
החזיקו את הצ'קליסט הזה לידכם בכל פעם שאתם יוצרים או משתמשים במפתח. עברו עליו עבור כל מפתח שברשותכם — הדפיסו אותו, או שמרו אותו ב-Notion.

## ארבעת חוקי הזהב
- [ ] **גבול קשיח (Hard Limit)** — תקרת הוצאה מוגדרת מראש על כל מפתח בתשלום. אצלנו היא מובנית: קרדיט שנטען מראש, בלי טעינה אוטומטית מחדש (Auto-reload) — אי אפשר לחייב מעבר למה שטענתם.
- [ ] **בידוד (Isolation)** — מפתח נפרד לכל פרויקט, כך שדליפה של מפתח אחד דורשת מכם לבטל רק אותו.
- [ ] **רוטציה (Rotation)** — החליפו מפתחות בערך כל 90 יום, או בכל פעם שמישהו בעל גישה עוזב.
- [ ] **הגבלות (Restrictions)** — הגבילו כל מפתח לשירות או לכתובת ספציפיים (לדוגמה, הגבלה ל-Gemini API).

## איפה מפתח אף פעם לא צריך להיות (אזורי סכנה)
- [ ] לא בתוך הקוד או ב-GitHub — בוטים סורקים מאגרים ציבוריים (repositories) תוך שניות. (הקוד משתמש במשתנה סביבה או בקובץ `.env`, אף פעם לא במפתח עצמו בתוך הקוד).
- [ ] לא בצ'אט או באימייל — WhatsApp, Slack, Teams, אימייל וכרטיסי תמיכה (support tickets) משאירים עותקים גלויים, אפילו לנציג רשמי של החברה.
- [ ] לא בכלי צד שלישי שאינכם סומכים עליו — הדבקת מפתח מעניקה למפתח הכלי גישה מלאה לחשבון שלכם. שאלו קודם: האם לכלי יש דף אבטחה, והאם הוא שומר מפתחות בצורה מוצפנת?

## שמירה נכונה של המפתח
- [ ] שמור ב-Bitwarden בתור Note (ולא בתור Login).
- [ ] לרשומה שם ברור שאומר איזה מפתח ולאיזה פרויקט, ושיוך לתיקייה.
- [ ] המפתח עצמו יושב בשדה מוסתר (Hidden) בשם key — לא בטקסט גלוי.
- [ ] כדי לשתף, השתמשו ב-Bitwarden Send (חד-פעמי) או ב-Collection משותף (צוותי) — אף פעם לא ב-WhatsApp או באימייל.

## אם מפתח דולף
- [ ] **ביטול (Revoke)** — מחקו את המפתח בלוח הבקרה (AI Studio / OpenAI). הגישה תיחסם תוך שניות.
- [ ] צרו מפתח חדש ועדכנו את הכלים שלכם.
- [ ] בדקו את יומני השימוש (usage logs) כדי לזהות פעילות לא מורשית.

> ביטול מפתח הוא פעולה זולה, מהירה ואינה מהווה כישלון — זוהי בדיוק היכולת שמאפשרת לכם לעבוד בביטחון.

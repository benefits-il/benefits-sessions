# סיכום · MCP — החיבור האוניברסלי

## מה שלמדנו

1. **הבעיה: כאוס האינטגרציה.** כל חיבור בין מודל לכלי דרש קוד ייעודי משלו — חמישה מודלים כפול עשרה כלים שווה חמישים חיבורים לתחזק. זו בעיית N על M.
2. **הפתרון: MCP הוא ה-USB של ה-AI.** ‏Model Context Protocol — תקן פתוח אחד שכל מודל וכל כלי מדברים. מחברים מקור מידע או כלי פעם אחת, וכל AI שתומך בתקן משתמש בו מיד.
3. **שלושה תפקידים בכל חיבור:** ‏Host — האפליקציה שעובדים בה (אצלנו קלוד) · ‏Client — המתורגמן שמנהל את הפרוטוקול בתוכה · ‏Server — התקע שמתחבר לכלי ומשדר אותו פנימה.
4. **MCP מול API: הקיוסק מול מכונת החטיפים.** ‏API רגיל הוא מכונת חטיפים בלי חלון זכוכית — לא רואים מה בפנים ואין את מי לשאול; שרת MCP הוא הבחור הנחמד בקיוסק השכונתי — יודע בדיוק מה יש אצלו ומכין לפי בקשה. קלוד מגלה לבד אילו כלים יש, מבין מה הם עושים ומבקש בדיוק את מה שהוא צריך.
5. **MCP מול קונקטור מוכן: בעלות מול נעילה.** קונקטור נוח אבל נעול לפלטפורמת הספק; ‏MCP שלכם ונייד. בונים משלכם רק כשיש סימן אמיתי: לוגיקה ייחודית · צורך בניידות · תהליך מורכב · שליטה ואבטחה. אחרת — קונקטור מוכן מנצח, ולפני הכול בודקים אם כבר קיים MCP מוכן.
6. **חיברתם MCP ראשון.** ‏NotebookLM — קליק אחד, login של גוגל, בלי מפתח — וקלוד גילה את הכלי והשתמש בו לבד. בכיתה: בונים MCP משלכם.

|---|---|---|---|
| גילוי יכולות | דינמי — המודל מגלה לבד | מוגדר מראש בקוד | מוגבל לפונקציות הספק |
| ניידות | אוניברסלי — כל כלי תומך | פרטני לכל חיבור | נעול לפלטפורמה אחת |

# תדריך-פתיחה: בניית System Prompt למטא-סוכן

> **איך משתמשים:** הדבק את כל הטקסט הזה ל-AI חזק (Claude וכד'). הוא יריץ ראיון קצר ואז יפיק עבורך **system prompt אחד, שלם ועצמאי, למטא-סוכן** — סוכן שבונה סוכנים. אל תענה מראש על השאלות; תן ל-AI לשאול אותך.

---

אתה הולך לבנות עבורי **system prompt למטא-אג'נט** — סוכן שתפקידו לבנות סוכנים אחרים, skills, ותדריכים שמנחים מערכות אחרות לבנות סוכנים. קרא את כל התדריך לפני שאתה מתחיל, ואז עקוב אחרי התהליך למטה.

## ההקשר שלי

עברתי deep research מעמיק על איך בונים סוכני-AI אמינים בעולם. אני מכיר את הדפוסים (ReAct, Plan-and-Execute, Reflection, Orchestrator-Worker), את מתמטיקת הצטברות-השגיאות, ואת הקונצנזוס ש"מתחילים פשוט ומוסיפים מורכבות רק כשמוכח". אתה לא צריך ללמד אותי את היסודות — אתה צריך לבנות לי כלי מצוין. אפשר ללכת לעומק.

## מה אני רוצה שתבנה

system prompt **יחיד ועצמאי** (לא חבילת-קבצים, לא skill) שניתן להדביק בכל פלטפורמה. המטא-סוכן שייווצר ממנו חייב:

- **להיות אדפטיבי** — לרוץ ולייצר עבור Claude web, Cowork, Claude Code, וגם סוכני-רשת / API. האנטומיה קבועה; המימוש של Tools/Knowledge/Memory/פורמט-פלט משתנה לפי הפלטפורמה.
- **לייצר חמישה סוגי תוצרים:** הוראות סוכן ל-web, הגדרת Cowork, חבילת Claude Code (SKILL.md / CLAUDE.md / subagents), מפרט סוכן-רשת/API, **ובריף-הנחיה למערכת אחרת**.
- **לדעת לבנות גם skills** כתוצר בפני עצמו.
- **לעשות גילוי כלים אמיתי** — בשלב הכלים, לחפש MCP/קונקטורים קיימים ולהציע אותם, לא להמציא שמות.
- **לעבוד צעד-אחר-צעד** עם chain of thought גלוי, וליצור מסמכים.
- **להחליט במפורש**, לכל סוכן שהוא בונה, האם צריך memory / knowledge / tools — וברירת המחדל לכל אחד היא **לא**, מתווסף רק כשמתקיים תנאי-הפעלה.

## התהליך שאני רוצה שתעקוב אחריו

**צעד 0 — ראיון קצר.** אל תשפוך עליי את כל השאלות. שאל אחת-שתיים בכל פעם, התאם לפי תשובותיי, ואל תשאל על מה שכבר ענה עליו התדריך הזה. מינימום שאתה חייב להבין: (א) מה התפקיד והגבולות של המטא-סוכן שלי, (ב) באילו פלטפורמות הוא חייב לתמוך, (ג) אילו סוגי תוצרים הוא חייב לייצר, (ד) **כמה רחוק אני רוצה ללכת ברמת האימות (evals)** — מבדיקת-שפיות איכותית ועד סט בדיקות מלא. את עומק האימות וכל שאר רמות-העומק נקבע יחד בראיון.

**צעד 1 — סיכום-כיוון לאישור.** הצג לי סיכום קצר (שם, תפקיד, פלטפורמות-יעד, סוגי-פלט, רמת-אימות). אל תמשיך עד שאאשר.

**צעד 2 — עיצוב מבוסס-מחקר.** עבוד לפי "research over assumption" — כשאתה מזהה צורך ביכולת, חפש את המימוש הטוב הקיים במקום להסתמך על הזיכרון:
- **כלים:** חפש MCP/קונקטורים אמיתיים (רישומים רשמיים, ספריות קהילה). לכל כלי כתוב חוזה: שם, פרמטרים, תיאור, דוגמה, תופעות-לוואי, התנהגות-בכשל.
- **זיכרון:** עץ-החלטה — צריך זיכרון בין סשנים? מעקב-התקדמות במשימה ארוכה? למידה מתיקונים? משותף עם סוכנים אחרים? או חסר-מצב לגמרי? הוסף רק את מה שנדרש, עם מדיניות מה נשמר ומה נזרק.
- **ידע:** הבחן בין ידע סטטי (בקונטקסט) לבין נשלף-בזמן-ריצה; אל תציף קונטקסט.
- **דפוס בקרה:** בחר (ReAct כברירת מחדל; Plan-and-Execute למשימות מורכבות), עם **תנאי עצירה** וטיפול-שגיאות — לעולם לא רצף לינארי נוקשה.

**צעד 3 — כתיבת ה-system prompt.** כתוב את המטא-סוכן כ-system prompt אחד. סדר הסעיפים = סדר עדיפות-הקשב של המודל: זהות → עקרונות → התנהגות (ראיון) → אדפטיביות-פלטפורמה → תהליך-בנייה → אנטומיית-הסוכן שהוא ממלא → גילוי-כלים → פורמטי-פלט → אימות → סוכן-יחיד-מול-רב-סוכני. נסח בחיוב **עם נימוק** (לא רק "אל תעשה"), כי הנימוק מונע מהסוכן למצוא פרצות.

> **מבנה וסימון (delimiters):** תחום את סעיפי ה-system prompt בתגי XML (למשל `<identity>`, `<principles>`, `<process>`) כשהיעד הוא Claude — זה מפריד הוראות מתוכן ומפחית דליפה בין סעיפים. אל תקנן מעל 2 רמות (קינון-יתר מוסיף רעש). הבחירה תלוית-יעד: ל-Claude (web/Cowork/Code) תגי XML; לסביבות אחרות / סוכני-רשת לרוב כותרות Markdown נקיות. **המטא-סוכן שתבנה צריך לבחור את הסימון לפי פלטפורמת-היעד** — בדיוק כמו שאר ההתאמות, ולא רק להשתמש בו ב-system prompt של עצמו.

**צעד 4 — מחסום-איכות לפני מסירה.** ודא, ותקן לפני שאתה מוסר:
- [ ] **בהירות-הפעלה** — מתי המטא-סוכן פועל ומתי לא.
- [ ] **least privilege** — רק הכלים/ההרשאות שבאמת נדרשים; פעולות הרסניות דורשות אישור.
- [ ] **חוזה-עבודה** — יש שלב-תכנון, שלב-אימות עם קריטריון-הצלחה, ומסלולי התאוששות-משגיאה.
- [ ] **כלכלת-קונטקסט** — בלי הוראות גנריות ("כתוב קוד טוב"), בלי כפילויות.
- [ ] **נאמנות-פרסונה** — הפרסונה מספיק ספציפית כדי להניע החלטות אוטונומיות; אני לא אצטרך להסביר מחדש את המשימה אחרי ההפעלה.
- [ ] **בדיקת אנטי-דפוסים** — בלי מסגור "עוזר כללי", בלי קינון XML עמוק, בלי בלוקי-קוד שצריכים להיות הפניות.

## עקרונות מנחים למטא-סוכן שתבנה

מתחילים פשוט — סוכן יחיד כברירת מחדל; רב-סוכני רק כשמוכח שיחיד נכשל, ואז orchestrator + תת-סוכנים מבודדים שמחזירים סיכומים, כתיבה חד-חוטית. research over assumption. progressive disclosure. פלט מודע-סביבה. אף פעם לא להסתמך על יכולת/כלי שלא אומתו שקיימים ביעד.

## פורמט הפלט הסופי שלך אליי

מסור **system prompt אחד**, קריא וניתן-לעריכה, בשפת העבודה שלי (עברית, מונחים טכניים באנגלית), אבל שהמטא-סוכן יידע לייצר סוכנים בכל שפה. בסיום: הסבר בקצרה את החלטות-העיצוב המרכזיות, והצע לי הרצת בדיקת-שפיות על משימה לדוגמה.
